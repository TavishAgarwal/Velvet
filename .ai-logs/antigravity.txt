## THE APP — VELVET

**Brand:** Dark luxury editorial. Deep blacks, warm gold (`#C9A84C`), Playfair Display headings, DM Sans body. Think a private members' club at night.  
**Tagline:** "Not everyone gets in."  
**Reference:** Lox Club (loxclub.com)

### Core User Flows
1. **Application** → Multi-step form → Submission → Wait for approval
2. **Admin Review** → See queue → Read essay → Approve / Reject / Waitlist → Email sent
3. **Onboarding** → Approved member sets avatar + bio → Enters the app
4. **Member Daily Use** → Home feed → Browse directory → View profiles → RSVP events → DM members
5. **Invite** → Generate codes → Share → Track who joined

### Roles
| Role | DB Value | Access |
|------|----------|--------|
| Applicant | `applicant` | Apply only |
| Pending | `pending` | Applied, awaiting review |
| Member | `member` | Full member access |
| Admin | `admin` | Member access + admin dashboard |

## CRITICAL RULES — NEVER VIOLATE

### Design
- **Never hardcode hex values.** Always import from `lib/theme.ts`
- **Never use raw `<Text>`.** Always use `components/ui/Text.tsx` with a `variant` prop
- **No spinners for content loading.** Use `SkeletonLoader` shimmer cards
- **All screens** have: loading skeleton, empty state, error state with retry
- Background on every screen: `BG_BASE (#0A0A0A)`
- Primary CTA is always `GoldButton` — gold bg, black text

### Code Quality
- TypeScript strict mode — no `any` types anywhere
- All shared types in `types/index.ts` (see COMPONENTS.md for full interface list)
- Import colors only from `lib/theme.ts`, fonts only from `lib/typography.ts`
- Every TanStack Query hook uses `placeholderData` from `lib/mockData.ts`
- Never use `useEffect` for data fetching — use TanStack Query hooks

### Database & Auth
- RLS policies must be set on every table before testing
- Never trust client-side role checks alone — Supabase RLS is the real guard
- `profiles.role` is the single source of truth for access control
- Always check `enabled: !!id` on single-entity queries to prevent null fetches

### Navigation
- Auth guard lives only in `app/_layout.tsx` — nowhere else
- Use `router.replace()` for auth redirects (no back navigation to auth screens)
- Use `useLocalSearchParams<{ id: string }>()` for all dynamic route params

### Realtime
- Enable Replication in Supabase dashboard for: `messages`, `notifications`, `applications`
- Always clean up Supabase channel subscriptions in `useEffect` return function
- Deduplicate Realtime inserts before adding to TanStack Query cache

## COMMON GOTCHAS — READ BEFORE BUILDING

1. **Fonts not loading** → Always `await useFonts()` and keep `SplashScreen` visible until loaded
2. **RLS blocking queries** → Check policies for each table; member queries require `role = 'member'`
3. **Realtime not firing** → Enable replication on table in Supabase: Database → Replication
4. **Image uploads failing** → Storage bucket policies must allow public read
5. **`useLocalSearchParams` undefined** → File must be at exact Expo Router path e.g. `app/member/[id].tsx`
6. **Android keyboard** → `behavior="height"` on Android, `behavior="padding"` on iOS
7. **Home indicator overlap** → Always use `useSafeAreaInsets().bottom` for fixed bottom buttons
8. **Duplicate Realtime messages** → Check for existing message ID before appending to cache
9. **Conversations `OR` query** → Use `.or('member_1_id.eq.X,member_2_id.eq.X')` pattern
10. **NativeWind v4** → Requires `global.css` import in root `_layout.tsx`
11. **Reanimated plugin** → Must be LAST plugin in `babel.config.js`
12. **`placeholderData` not showing** → Must be exported from `lib/mockData.ts` and passed correctly

## PHASE 3 — DEAD END ELIMINATION
## Find and fix every navigation dead end in the app.

A dead end is any state where:
- A button does nothing when tapped
- A navigation action leads to a screen that crashes
- A screen has no way to navigate away from it
- A flow reaches a terminal state with no clear next action
- An error state has no retry or recovery path

### Check every interactive element:

Go through every screen and list every Button, TouchableOpacity, Pressable, 
Link, and Tab that has an onPress or href. For each one verify:

1. The handler is defined (not undefined, not empty function)
2. The destination route exists as a file in app/
3. The destination screen does not immediately crash on load
4. If the action is async, loading and error states are handled
5. After the action completes, the user is in a logical place

### Specific dead ends to look for and fix:

#### Application Flow Dead Ends
- Back button on step1 → should go to / (landing) not crash
- Back button on submitted screen → should not allow going back to step4
- Submit fails → user stuck on step4 with no feedback → fix: show error toast
- Referral code API call fails → should fail silently, not block submission

#### Auth Dead Ends
- Magic link expired → user lands in app with invalid session → 
  should redirect to /(auth)/login with message
- OAuth provider cancels → app should return to login screen, not hang
- Session refresh fails mid-session → should sign out gracefully 
  and redirect to login, not show broken data

#### Member Directory Dead Ends
- Member has no avatar and no display_name → initials would be "?" → 
  handle gracefully
- Tapping own profile in directory → should show own profile or redirect 
  to /(tabs)/profile, not show "Send Message"
- Search returns zero results → empty state shown, clear filter button works

#### Messaging Dead Ends
- Sending a message when OpenAI is down → fallback reply used, 
  not infinite spinner
- Starting conversation with a member who has been deleted → 
  handle gracefully, show error
- Thread screen opened with invalid conversation id → 
  show error state with back button
- Keyboard open, user taps back → keyboard should dismiss, 
  navigation should still work

#### Admin Dead Ends
- Admin approves last application in queue → queue shows empty state, 
  not broken UI
- Admin tries to approve an already-approved application → 
  action disabled or shows appropriate message
- Edge function call fails during approval → 
  what happens? DB may be updated but email not sent →
  fix: show warning toast that email failed, allow retry

#### Event Dead Ends
- RSVP toggled while offline → optimistic update applied, 
  then error occurs → must revert correctly
- Event detail opened for an event that has been deleted → 
  show not found state with back button
- Going filter selected when user has no RSVPs → empty state shown

#### Invite Dead Ends
- Generate invite tapped when invite_count is 0 → button should be disabled
- Copy to clipboard fails on certain devices → 
  show error toast, do not silently fail
- Share sheet cancelled by user → no error shown, stays on invites screen

#### Profile Dead Ends
- Edit profile save fails → user stays on edit screen, 
  error toast shown, data not lost
- Avatar upload fails → profile save should still proceed with 
  existing avatar_url, or show specific upload error

#### Onboarding Dead Ends  
- User closes app mid-onboarding → returns to onboarding on reopen 
  (onboarding_completed still false)
- Image upload fails → user should be able to skip avatar and complete 
  onboarding without photo
- Complete Profile tapped with no display_name → button disabled, 
  field highlighted with error

CREATE OR REPLACE FUNCTION public.review_application(
  app_id     uuid,
  new_status text,
  notes      text DEFAULT NULL
)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_user_id uuid;
BEGIN
  -- Only admins can review applications
  IF NOT is_admin() THEN
    RAISE EXCEPTION 'Forbidden: admin role required';
  END IF;

  -- Validate status value
  IF new_status NOT IN ('approved', 'rejected', 'waitlisted') THEN
    RAISE EXCEPTION 'Invalid status: %', new_status;
  END IF;

  -- Fetch the application (must be pending)
  SELECT user_id INTO v_user_id
  FROM public.applications
  WHERE id = app_id AND status = 'pending';

  IF v_user_id IS NULL THEN
    RAISE EXCEPTION 'Application not found or not in pending state';
  END IF;

  -- Update the application record
  UPDATE public.applications
  SET status      = new_status,
      admin_notes = COALESCE(notes, admin_notes),
      reviewed_by = auth.uid(),
      reviewed_at = now()
  WHERE id = app_id;

  -- If approved, promote the user to member and migrate ALL application data
  IF new_status = 'approved' THEN
    UPDATE public.profiles p
    SET role = 'member',
        display_name = COALESCE(p.display_name, a.full_name),
        city = COALESCE(p.city, a.city),
        profession = COALESCE(p.profession, a.profession),
        company = COALESCE(p.company, a.company),
        linkedin_url = COALESCE(p.linkedin_url, a.linkedin_url),
        instagram_handle = COALESCE(p.instagram_handle, a.instagram_handle),
        onboarding_completed = true
    FROM public.applications a
    WHERE a.id = app_id
      AND p.id = a.user_id;
  END IF;
END;
$$;
Once that's executed, newly approved members will seamlessly transition into the app with fully populated profiles!

Those are classic React/Expo Router race conditions! Here is exactly why they were happening and how I just fixed them:
1. The "Empty Dashboard" Sign-Out Bug
The Problem: When you clicked the Sign Out button, two things were trying to navigate you simultaneously. Our AuthContext was calling router.replace('/') at the exact same millisecond that the master route guard in _layout.tsx detected you were unauthenticated and also tried to fire router.replace('/'). Expo Router panics when it receives duplicate simultaneous routing commands, causing it to abort the navigation entirely and leave you stranded on a broken, data-less version of the dashboard you were just on. The Fix: I removed the manual redirect from the Sign Out button's code and deferred all routing exclusively to _layout.tsx so there is only one "single source of truth".
2. The Application Submit Routing Bug
The Problem: When you hit submit on Step 4, your status in the remote database immediately became 'pending'. However, _layout.tsx hadn't finished fetching this new status yet (it still thought your status was 'none'). When the app tried to jump to the /apply/submitted screen, the route guard instantly kicked in, saw you were trying to access the submitted screen with a status of 'none', and forcibly kicked you back to the beginning of the application (/apply). The Fix:
1. I decoupled the routing from the submit button. Now, the submit button just tells the database you are done and emits a applicationStatusChanged event behind the scenes.
2. _layout.tsx hears this event, successfully pulls your new 'pending' status from the database, and safely routes you to /apply/submitted on its own terms.
3. Finally, I wrapped all the routing commands inside _layout.tsx inside a setTimeout(..., 0) block. This is a crucial best-practice in Expo Router that tells the system to wait for React to finish rendering before it jumps to a new screen, guaranteeing that it never gets stuck or interrupts itself.
Both of these flows should now be completely bulletproof. Try fully reloading the app one last time and giving the application and sign-out flows a final test run!
